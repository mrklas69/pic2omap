# Sezení 2

Pokračování téhož dne. Cíl: Fáze 0 → začít databází symbolů (zadání uživatele).

## Diskuse

- Adresář přejmenován na `Pic2Omap/`, memory dir přesunut — hotovo mimo sezení.
- **Strategie symbolové DB**: namísto manuálního psaní YAML z ISOM PDF (200+ symbolů, riziko překlepů) extrahovat z existujících OMAP souborů. OMAP je XML s plnou definicí symbolů uvnitř — jediný zdroj pravdy, žádná divergence vůči OOM.
- **Architektura ve 2 vrstvách**:
  - Vrstva 1 (`SymbolLibrary`) = rich 1:1 model z OMAP
  - Vrstva 2 (`ColorProfile`, později `SymbolProfile`) = derivovaný model pro detector
- **Paralelní agenti** verifikovali oba OMAP soubory proti IOF PDF spec (ISOM 2017-2 + ISSprOM 2019-2).
- **Klíčové zjištění**: oba OMAP soubory používají **ISOM 2000 / OOM-default schéma**, ne IOF 2017-2 ani ISSprOM 2019-2. Číslování symbolů je posunuté o 1-3 pozice, některé prohozené. Stejný kód = různý symbol napříč spec verzemi (501 = Road v ISOM 2000/2017-2, Paved area v ISSprOM 2019-2).
- **Color separation strategy**: palette-based (fixní paleta z library, nearest in LAB), ne K-means. Deterministické, žádný unsupervised clustering.
- **Anti-aliasing problém**: vrstevnice se rozsekly mezi Brown a OpenOrienteering Orange (LAB distance malá). Řešení: `ColorCategory` vrstva — sloučit blízké barvy do sémantických rodin (BROWN, GREEN, BLACK, YELLOW, BLUE, PURPLE, WHITE, GRAY, RED) podle HSV hue.

## Rozhodnutí

- **DB schéma**: OOM-native, ne IOF přepisování. Pic2Omap bude pracovat s kódy, jak jsou v OMAP (širší kompatibilita s existujícími mapami).
- **Jazyk**: Python (potvrzeno, stdlib + numpy + opencv + skimage).
- **Symbol library**: extrahovat z OMAP parserem, ne ručně z PDF.
- **Color separation**: palette-based, ne K-means.
- **Color category**: HSV hue klasifikace + manuální NAME_OVERRIDES pro hraniční případy (Green 50% Yellow patří k GREEN, ne YELLOW).
- **Spec PDF**: stáhnout pro offline referenci do `docs/`, ale **negitovat** (22 MB, public download). Gitignore.
- **První git commit**: po dokončení dnešního sezení (~1100 řádků kódu + reports).

# Sezení 3

Pokračování téhož dne. Cíl: **Stage 3 — per-color raster ops** (morfologie + connected components + skeletonizace).

## Diskuse

- **Strategie morfologie**: konzervativní default `opening 2×2` pro všechny barvy, žádné `closing`. Důvod: vrstevnice na renderu jsou ~1px tlusté, agresivnější kernel by je smazal. Closing skrývá problémy, neřeší — vrátíme se k němu po vizuálním zhodnocení.
- **Klasifikace komponent na POINT/LINE/AREA**: iterativní ladění heuristiky.
  - v0: `area + density` → fragmenty vrstevnic mylně klasifikovány jako POINT.
  - v1: přidána `POINT_MIN_DENSITY=0.4` → posun 55 fragmentů z POINT do LINE.
  - v2 (finál): přidán `POINT_MAX_ASPECT=2.0` (kulatost / čtvercovitost bboxu) → posun dalších 83 komponent z POINT, většina (78) do AREA, 5 do LINE. POINT teď vyžaduje **smallness AND compactness AND kulatost** — tři podmínky.
- **WHITE problém**: pozadí (open forest 401) má v rasteru rozsekané komponenty (bbox skoro celá mapa, nízká density) → mylně klasifikováno jako LINE. **Řešení**: `split_category_masks` WHITE skipuje. Pokud se v budoucnu objeví potřeba detekovat open forest jako AREA symbol, řeší se to separátně.
- **Skeleton brown tloušťka 3×**: vrstevnice na rendered PNG mají efektivně 3 pixely (po 2×2 opening). Ne chyba, jen kalibrace na DPI. Není potřeba řešit teď.
- **AREA leak**: krátké tlusté linkové fragmenty (density > 0.5) se schovaly v AREA. Stage 4 bude potřebovat AREA min-size filter, aby je odlišil od ploch.

## Rozhodnutí

- **Klasifikační prahy** (kalibrované pro forest sample 631×478, ~300 DPI render):
  - `MIN_AREA = 4` (NOISE)
  - `POINT_MAX_AREA = 30`, `POINT_MIN_DENSITY = 0.4`, `POINT_MAX_ASPECT = 2.0`
  - `AREA_MIN_DENSITY = 0.5`
  - Per-DPI škálování zatím není (TODO).
- **WHITE skip** v components/skeleton (nikoli v morfologii — morfologie WHITE prostě nezmění, params `open_kernel=0`).
- **Morfologie konzervativní**: `opening 2×2` pro všechny barevné kategorie, žádné closing.
- **OpenCV** pro morfologii a components, **scikit-image** pro skeletonize (Zhang-Suen — robustnější než morphological thinning).
- **Stage 3 MVP cíl splněn**: `cat_brown_skeleton.png` = čisté 1px středovky vrstevnic, připraveno pro Stage 5 vektorizaci.

## Kód

Nové Python moduly (~430 řádků + komentáře):

- `morphology.py` — `MorphParams` dataclass, `CATEGORY_PARAMS` dict, `clean_mask`, `clean_category_masks`, IO helpery, `format_stats_report`.
- `components.py` — `ComponentType` enum (POINT/LINE/AREA), `ComponentInfo` dataclass s `density` a `aspect_ratio` props, `classify_component`, `extract_components` (cv2 wrapper), `split_by_type` (LUT-based vectorizace), `split_category_masks` se skipem WHITE.
- `skeleton.py` — `skeletonize_mask` (bool roundtrip kolem skimage), `skeletonize_line_masks`, IO, `format_skeleton_report` (počítá poměr LINE px / Skel px jako proxy tloušťky).
- `stage3_demo.py` — CLI orchestrátor: morphology → components → skeleton, `_stage3_report.txt`.
