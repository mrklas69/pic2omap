# 2026-05-20 — Sezení 9

Pokračování po Sezení 8. Cíl: db2omap "Příště" bod 1 (přesná georef), pak bod 2
(line vektorizace path-tracing). Oba dokončeny.

## Diskuse

- **OMAP coord je paper-space, ne reálný svět.** Object coords = pozice na
  papírové mapě v 1/1000 mm; PNG je render téhož papíru. Takže pixel → OMAP coord
  je čistá podobnostní transformace (scale + translace + y-flip), **BEZ rotace**.
  Rotace (declination/grivation), UTM a `.pgw` jsou potřeba jen pro reálný svět.
  Tohle zjednodušilo celý georef problém. Memory `georef-paper-space-not-world`.
- **Bbox-fit byl principiálně chybný.** Mapoval (0..png) → bbox template OBJEKTŮ,
  ale bbox objektů ≠ rozsah obrazu (mapa má vpravo prázdno, PNG má title).
  Empirie: Slovanka full y-osa sedí s `.pgw` (0.3176 vs 0.3175 m/px), ale x ne
  (0.2866) — protože objekty nevyplní šířku. Anizotropní protažení.
- **Scale plně určen DPI renderu**: `units_per_px = 25400 / DPI`. Ověřeno 2 zdroji
  shoda na 5 cifer: pHYs 1200 DPI → 0.31750 m/px ≡ `.pgw` √(A²+D²) = 0.31750.
- **forest sample.png je kontaminovaný** — prošel Paint.NET (tEXt tag), který
  přepsal pHYs na falešných 96 DPI (skutečné ~289), `.pgw` nemá. Nevhodný jako
  georef kalibrační cíl → zůstává na bbox-fit fallbacku. Memory `png-dpi-pitfalls`.
- **1:10000 vs scale=15000 nesoulad** — titulek Slovanky říká 1:10 000, ale OMAP
  `<georeferencing scale="15000">` i `.pgw` konzistentně implikují 1:15 000.
  Pro paper-space coords (otevření v OOM) nevadí; pro reálné UTM ano. K ověření
  uživatelem (zastaralý titulek nebo export v 1:15000).
- **Line vektorizace: greedy walk nestačí.** Topologie: 1923 line objektů, 94 %
  jednoduché křivky, 6 % junction. ALE: oněch 6 % = **15 % celkové délky** linií
  (junction objekty jsou nejdelší). Naivní greedy (1. větev) prošel u obj 1864
  jen 7 bodů z 2093 px. Změna strategie na segment-trace. Memory `skeleton-median-misleads`.

## Rozhodnutí

- **Rigorózní georef přes složení afinních matic**: `pixel --[.pgw]--> UTM
  --[projectedToMap]--> OMAP coord`. Rotace grivation z `.pgw` a z projectedToMap
  se ve složení vyruší → výsledek scale+translace. Skládám matice (numpy), neřeším
  vyrušení ručně.
- **Znaménková konvence ověřena empiricky** (4 varianty θ=±grivation, flip=±):
  vítěz `MapToProj = proj_ref + scale·1e-6 · R(-grivation) · diag(1,-1) · map`.
  Test: 100 % GT coords uvnitř PNG, y vyplní výšku; 92.9 % centroidů na ne-bílém
  pixelu; export bbox ≈ GT bbox (99 %); vizuální overlay sedí na struktury mapy.
- **Fallback řetězec**: `.pgw` + georef když je world file a CRS není Local, jinak
  bbox-fit (forest sample). MapToProj invertuju numericky (vyhne se chybám znaménka).
- **Line vektorizace = segment-trace** (uživatel zvolil možnost C po revizi).
  Kostra jako graf: uzly = deg≠2 (endpoint/junction), hrany = řetězy deg-2 pixelů.
  Každá hrana = samostatný OMAP path se stejným symbolem. Neztratí délku.
  Limitace v1: L-rohy v 8-souvislé kostře sekají segment v ohybu (víc kratších
  paths, délka zachována). Bezier + re-linking = samostatné TODO body.
